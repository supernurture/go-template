## Overview

`go-template` is an opinionated starter template for building RESTful APIs in Go. It wires together a curated set of battle-tested libraries — **Gin** for routing, **pgx** for PostgreSQL, **sqlc** for type-safe SQL, **oapi-codegen** for OpenAPI-driven handlers, **Viper** for configuration, and **Zap** for structured logging — into a clean, layered architecture that scales from a weekend project to a production service.

Use this template as a baseline whenever you start a new Go backend service. Everything is pre-wired so you can focus on writing business logic instead of boilerplate.

## Project Structure

```
go-template/
├── api/
│   └── server/           # HTTP server setup and route registration
├── cmd/
│   └── api/              # Application entrypoint (main.go)
├── configs/              # Viper-based configuration loading
├── internal/             # Private application code
│   └── infrastructure/
│       └── database/     # Database connections, sqlc-generated queries, sqlc.yaml
├── pkg/
│   └── logger/           # Shared Zap logger wrapper
├── scripts/
│   └── oapicodegen.sh    # Shell script to run oapi-codegen
├── .env.example          # Sample environment variable file
├── Makefile              # Developer shortcuts
├── go.mod
└── go.sum
```

### Directory Responsibilities

**`cmd/api/`** — The application entry point. Initialises configuration, connects to the database, wires dependencies, and starts the HTTP server.

**`api/server/`** — HTTP server construction and route registration. Handlers generated by `oapi-codegen` are attached here.

**`configs/`** — Configuration structs and Viper loading logic. Settings are read from YAML/TOML config files merged with environment variables, enabling different configs per environment (`development`, `sit`, `uat`, `production`).

**`internal/`** — All code that should not be imported by external packages. The `infrastructure/database` sub-package holds the database pool setup, `sqlc`-generated query files, and the `sqlc.yaml` generation config.

**`pkg/logger/`** — A thin, reusable wrapper around Uber's Zap logger, ready to import from any layer.

**`scripts/`** — Shell scripts invoked by the Makefile. `oapicodegen.sh` drives the `oapi-codegen` tool to regenerate server stubs from an OpenAPI spec.

## Getting Started

```bash
# 1. Clone the repository
git clone https://github.com/yonathandj/go-template.git
cd go-template

# 2. Copy the example environment file
cp .env.example .env

# 3. Fill in your database credentials in .env (see Environment Variables below)

# 4. Download dependencies
go mod download

# 5. Run the server
go run ./cmd/api
```

## Environment Variables

Copy `.env.example` to `.env` and set the values before running the application.

| Variable | Description | Allowed Values |
|---|---|---|
| `APPLICATION_ENVIRONMENT` | Runtime environment | `development`, `sit`, `uat`, `production` |
| `GIN_MODE` | Gin framework mode | `debug`, `test`, `release` |
| `DATABASES_POSTGRE_ANALYTICS_USERNAME` | Username for the analytics PostgreSQL database | — |
| `DATABASES_POSTGRE_ANALYTICS_PASSWORD` | Password for the analytics PostgreSQL database | — |
| `DATABASES_POSTGRE_AUDIT_USERNAME` | Username for the audit PostgreSQL database | — |
| `DATABASES_POSTGRE_AUDIT_PASSWORD` | Password for the audit PostgreSQL database | — |

> **Tip:** Non-secret configuration (host, port, database name, pool sizes, etc.) is managed through Viper config files in `configs/` and can be committed safely. Only credentials and environment selectors belong in `.env`.

## Code Generation

This template relies on two code generators. Run them whenever you change your OpenAPI spec or SQL queries.

### OpenAPI → Server stubs (`oapi-codegen`)

```bash
make oapicodegen
```

This invokes `scripts/oapicodegen.sh`, which calls `oapi-codegen` to regenerate Go interfaces and request/response types from your OpenAPI specification. Drop your updated `.yaml` spec in the appropriate location, then re-run this command to keep generated code in sync.

### SQL → Type-safe queries (`sqlc`)

```bash
make sqlcgen
```

This runs `sqlc generate` using the configuration at `internal/infrastructure/database/sqlc.yaml`. After adding or modifying `.sql` query files, re-run this command to regenerate the corresponding Go functions and types.

|---|---|

## Architecture

The template follows a **layered / clean architecture** approach:

```
┌───────────────────────────────┐
│         HTTP Layer            │  api/server  — Gin routes, generated handlers
├───────────────────────────────┤
│       Application Layer       │  internal    — Use cases and domain logic
├───────────────────────────────┤
│      Infrastructure Layer     │  internal/infrastructure — DB, external services
├───────────────────────────────┤
│       Shared Packages         │  pkg         — Logger, utilities
└───────────────────────────────┘
```

Key design decisions:

- **OpenAPI-first** — define your API contract in YAML, generate the Go server interface with `oapi-codegen`, and implement against that interface. The spec is always the source of truth.
- **sqlc for database access** — write plain SQL, get compile-time-checked Go functions back. No ORM magic, no runtime surprises.
- **Two PostgreSQL databases** — the template is pre-wired for separate `analytics` and `audit` databases, a common pattern for segregating operational from audit/reporting data.
- **Environment-aware config** — Viper loads the appropriate YAML config based on `APPLICATION_ENVIRONMENT`, with `.env` layered on top for secrets.
- **Structured logging** — Zap is wired up from the start, encouraging structured, levelled logging from day one.
